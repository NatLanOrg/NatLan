Build artifacts

Output of compilation and linking

The graph

An artifact is a graph of four node types. Sections give the document its structure. Entities are the domain concepts. Requirements are EARS statements attached to entities. Relationships are typed edges between entities, and every edge is a product of a requirement.

Section

Document structure as a parent and child tree. Holds the verbatim text for reconstruction.

Entity

A domain concept with a stable id, a scope, and an internal or external linkage.

Requirement

An EARS statement attached to one or more entities. Behavior or constraint is derived from the pattern.

Relationship

A typed edge between two entities, reified, with a UML type and multiplicity.

  Section ──contains──▶ Section
     │
     │ declares
     ▼
  Entity ◀──attached to── Requirement ──implies──▶ Relationship
     ▲                                                  │
     └──────────────── connects ───────────────────────┘

  Diagnostic ──▶ flags any Entity, Requirement, or Relationship

The format

Artifacts are written as YAML, validated against JSON Schema. Diagnostics are first class and sticky: they carry their reasoning and are reconciled across builds rather than regenerated.

entities:
  - id: e_session
    name: Session
    scope: public
    linkage: external
    definition: A signed-in user's authenticated period of activity.

requirements:
  - id: r_session_expiry
    entities: [e_session]
    text: When 30 minutes pass with no activity, the session expires.
    ears:
      type: event
      trigger: 30 minutes pass with no activity
      response: the session expires

relationships:
  - type: association
    from: e_user
    to: e_session
    multiplicity: { from: '1', to: '0..*' }

diagnostics:
  - severity: warning
    code: contradiction
    reasoning: One document says sessions expire, another says they never do.
    entities: [e_session]

Schemas live under docs/compiler and publish at jazyk.org/schemas.
